## Section 4: Philosophy

### S12 — Why Not Just Commit Skills?
The naive approach: check skills into each project repo.
- Stale copies across repos — one update, N repos to patch
- Version matrix hell when skills ship scripts/binaries
- No shared updates, no team consistency
- **Punchline**: "Copying is easy. Keeping copies in sync is the real problem."

### S13 — Why Schools?
One source of truth for an entire organization.
- Single repo → symlinked into every project
- Edit in any project → propose back to the school
- New team member? `ace setup` and they have everything.
- Convention over configuration — do the obvious right thing automatically.

### S14 — Why Always-Latest?
The versioning philosophy that breaks from tradition.
- Traditional: dumb consumer, pin versions to prevent breakage
- LLM is not a dumb consumer — it reads, understands, adapts
- Same prompt → different output anyway. Pinning can't fix non-determinism.
- Symlinks to cache, not frozen copies
- **Bold claim**: "The consumer is intelligent. Always-latest is cheap."

### S15 — Dumb Sync, Smart Backend
ACE's design principle: do as little as possible.
- ACE = file sync + config merge + exec. That's it.
- No skill resolution, no dependency graph, no runtime logic
- Defer everything to the backend — the LLM is smarter than any resolver
- If the backend can do it, ACE shouldn't.
- **Punchline**: "The best orchestrator does the least orchestration."

## Section 5: Schools & Skills

### S16 — What is a School?
A git repo containing skills, conventions, and config for an org.
- Structure diagram: `school.toml`, `skills/`, skill folders
- One school → many projects (frontend, backend, etc.)

### S17 — Skills
Modular packages that extend AI capabilities.
- Specialized workflows, tool integrations, domain expertise
- Examples: `rust-coding`, `typst-coding`, `general-coding`
- Skills are loaded contextually — AI reads them, adapts.

### S18 — Importing Skills
Not everything lives in your school — import from the ecosystem.
- `ace import anthropics/skills --skill commit`
- skills.sh — Anthropic-blessed community skill registry
- No `npx skills` needed — `ace import` handles it directly
- Imported into the school, not just your machine — whole team gets it

## Section 6: Architecture

### S19 — Three Layers
Three-column layout:
1. **Config** — Dumb I/O. Reads files, no logic.
2. **State** — Domain tree. Merge, resolve, run actions.
3. **Ace** — Entrypoint. Holds state, drives CLI.
- Dependency arrow: `Config ← State ← Ace`

### S20 — The Lifecycle
Numbered sequence (maybe animated with `#pause`):
1. Discover config → 2. Parse & merge → 3. Authenticate
4. Fetch school → 5. Sync skills → 6. Check tooling
7. Select backend → 8. Inject prompt → 9. Exec
- **Key point**: all of this happens transparently on every `ace` run.
